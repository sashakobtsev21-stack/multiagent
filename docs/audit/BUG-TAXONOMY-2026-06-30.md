## C1 — Контаминация форка движком Грузии

**Симптом.** Новый сайт-форк тащит грузинские артефакты на трёх слоях: (1) `.claude/agents/*` и `.claude/skills/*` подписаны «Georgia Guidebook», translator-агенты ждут несуществующий ru-исходник (на en-only форке `/content` плодит ru/грузинский брак или падает на парности); (2) `scripts/*.mjs` шлют чужой `User-Agent`/email (`GeorgiaGuidebook`, `info@georgiaguidebook.com`) во внешние запросы за фото (Wikimedia/Openverse/Unsplash); (3) JSON-LD schema c `addressCountry: "GE"` в layout-ах (`ArticlePage`/`CityFoodPage`/`EdaDirectory`), грузинский `TITLE_SUFFIX` в `qa.mjs`, грузинские слаги/реалии (GEL, Сарпи, EKTA) в `SPEC/HANDOFF/PROGRESS`.

**Как детектить.**
```
git -C <repo> grep -rIl -iE "georgiaguidebook|Georgia Guidebook|addressCountry.*\"GE\"|GEL|Сарпи" -- src .claude scripts
```
Проверено 2026-06-30: в боевом `src` форков чисто (ME — только комментарий «форк движка Georgia Guidebook» в `content.config.ts`, не баг). В `.claude` AL/MK остался 1 хит в `fact-checker.md` — и это **корректно**: агенту прямо предписано ФЛАГировать грузинские артефакты (GEL/Сарпи/Georgia) в en-материале. Схема в Хорватии чинена коммитом `61c5f2f` (`addressCountry GE→HR` в 3 layout-ах) — для нового форка это обязательный шаг.

**Как НЕ допустить (форк-чеклист, до первого `/content`).**
- Прогон `grep` выше = 0 хитов в `src` (кроме комментария происхождения) и в `qa.mjs` (`TITLE_SUFFIX`), и в `scripts/*` (UA/email).
- `addressCountry` в layout-ах = код страны нового сайта (BA/AM/RS).
- `.claude/agents/*` + `.claude/skills/*` перебрендированы под страну; для en-only — удалить `uk-translator`, переписать `en-translator` под native-en (не ru→en); `seo-content-writer` не должен быть инструктирован «писать ru».
- Образец чистого форка — Черногория/Хорватия (1–2 остаточных хита). Привязка: `check-links` ловит битые грузинские слаги при сборке; парность ловит ложные ru на en-only.

## C2 — Прямые affiliate-URL вместо `/go/` (EKTA + болванки)

**Симптом.** Партнёрская ссылка в контенте или в `partners.json` ведёт прямым URL без атрибуции вместо трекинг-ссылки через редирект `/go/{partner}`. Каждый клик по самой денежной (YMYL-страховой) странице = €0. Подтверждено в `partners.json` всех 5: `"ekta": { "urlTemplate": "https://ektatraveling.com/", "allowSubId": false }`; у Черногории вдобавок дремлющие болванки `discovercars`/`getyourguide`/`booking` → прямые `https://www.*.com/`, `allowSubId:false`.

**Как детектить.**
```
git -C <repo> grep -nE "\"urlTemplate\":\s*\"https?://(?!.*\{subid\})" -- src/data/partners.json   # прямой URL без {subid}
git -C <repo> grep -rIlE "https?://(ektatraveling|discovercars|getyourguide|booking|safetywing)\.com" -- src/content  # прямой партнёр-URL в теле статьи (ЗАПРЕЩЕНО)
```
Проверено 2026-06-30: EKTA и др. **более не отрендерены ни в одной статье** (прямые `/go/ekta` и боксы сняты — GE `a561988`, ME `cdfcb15`, HR `5e1d9be`); утечка в partners.json остаётся «спящей» до тех пор, пока кто-то не впишет `partner:'ekta'` в новую статью.

**Как НЕ допустить.** Привязка: `check-links` проверяет, что `/go/{partner}` резолвится в `partners.json` (иначе клик в фолбэк `/` — предупреждение). Правило (зафиксировано в `$schema-note` partners.json и SPEC §16): **прямые URL партнёрок в контенте запрещены — только `/go/`**. Для нового сайта: не публиковать денежный кластер (страховка/авто/экскурсии/отели), пока соответствующий партнёр в `partners.json` не имеет рабочего `?sub_id={subid}` + `allowSubId:true`. Болванку с прямым URL не вписывать в статью даже временно (память: «впишут partner до фикса URL → 100% кликов без атрибуции»).

## C3 — Slug-коллизии / контент-каннибализация (одна интент-страница)

**Симптом.** Два класса: (а) точная коллизия `slug` внутри одной категории (сломанная маршрутизация/дубль файла); (б) две почти-одинаковые страницы под один поисковый интент каннибализируют друг друга (напр. дубль `borjomi` достопримечательность+город в Грузии; обратные транспортные пары `X-to-Y` и `Y-to-X`; itinerary-кластер 5 статей под один запрос). Исторически — худшая каннибализация была у Македонии (itinerary 5→1-2, transport/Tetovo 2→1).

**Как детектить.** Гейт `scripts/check-dedup.mjs` (RATCHET #3) ловит три класса по корпусу `.md` одного языка: (a) точные slug-коллизии в категории → ЖЁСТКИЙ FAIL без baseline; (b) обратные транспортные пары → FAIL если не в baseline; (c) почти-дубли заголовков — нормализованное равенство ИЛИ token-Jaccard ≥ 0.85 → FAIL если не в baseline. Baseline `scripts/.dedup-baseline.json` фиксирует принятые пересечения (slug-коллизии НЕ бейслайнятся).

**Как НЕ допустить.** Привязка: `check-dedup` (#3) — обязателен зелёным. Для нового сайта baseline `.dedup-baseline.json` стартует **пустым** (подтверждено решением владельца 2026-06-30) — значит ни одна коллизия/почти-дубль не должна появиться. STANDARDS §④ «Уникальность намерения (одна интент-страница)» — жёсткий гейт: один интент = одна страница, остальное folds/сателлит с canonical. Один город = одна статья `category=goroda`/cities (не плодить дубль «город» + «достопримечательность» под одним именем).

## C4 — Фото-дефицит (<5 инлайн / cover-only / вес >200КБ)

**Симптом.** Статья проходила старый гейт (была только проверка наличия cover ≤200КБ), но по факту cover-only или <5 инлайн-изображений → слабее ранжируется и конвертит, особенно в money-кластерах. Также: cover >200КБ роняет перф-бюджет.

**Как детектить.** Гейт `scripts/check-photos.mjs` (RATCHET #2) считает ВСЕ изображения (cover + gallery[] + inline `<img>`/markdown `![` + фото каждой остановки маршрута) и требует минимум: статья ≥5; новость ≥2; маршрут = cover + ≥1 фото на остановку (иначе ≥5). ERROR валит сборку для `articles`/`routes`; cover отсутствует/несуществует/>200КБ → ERROR. Baseline `scripts/.photo-baseline.json` grandfather-ит legacy-нарушителей.

**Как НЕ допустить.** Привязка: `check-photos` (#2) зелёным. Проверено независимо 2026-06-30: **0 статей под порогом** во всей сети → аудиторский P1-4 «фото-дефицит» закрыт. Для нового сайта `.photo-baseline.json` стартует пустым: каждая новая статья ОБЯЗАНА набрать ≥5 уникальных изображений (своё/CC/PD с атрибуцией) до публикации. STANDARDS §«Фото»: статья ≥5, новость ≥2, маршрут — каждая остановка, город ≥10, столица ≥15.

## C5 — Парность языков / сироты-переводы / lang↔папка

**Симптом.** На многоязычном сайте (GE ru/uk/en; ME/HR en+ru) одна языковая версия есть, парная отсутствует → «сирота-перевод»; или `lang:` во frontmatter не совпадает с языковой папкой; или числовые поля маршрута (days/distanceKm/budget) разошлись между языками. На en-only форках (AL/MK) — обратная проблема: ложные ru/uk-файлы от грузинского translator-агента.

**Как детектить.** Гейт `scripts/check-parity.mjs`: непарные ru/uk, рассинхрон lang↔папка, title > 60 символов, расхождение числовых полей маршрутов между языками. Exit 1 при нарушениях. SEO-мета считается в **символах, не байтах** (кириллица 2 байта/символ → ложные «над лимитом», память `seo-meta-length-chars-not-bytes`).

**Как НЕ допустить.** Привязка: `check-parity` зелёным. Для нового сайта: зафиксировать языковую политику в STANDARDS до контента (новые сайты — EN-first/en-only по решению владельца 2026-06-30). En-only форк не должен иметь ru/uk-папок и ru/uk-translator-агентов (см. C1). Публиковать языки атомарно (все версии одним коммитом), иначе сирота повиснет до следующего прохода.

## C6 — Внутренние ссылки: формат/префикс /ru/ ломает `test:links`

**Симптом.** Внутренняя ссылка без хвостового слеша или без языкового префикса на странице из дерева `/ru/` (`/uk/`) → битая ссылка при `trailingSlash:'always'` или утечка из языкового поддерева. `test:links` падает при публикации (память `astro-internal-link-format`: ссылка = `/{категория}/{slug}/` со слешем + `/ru/`).

**Как детектить.** Гейт `scripts/check-links.mjs` (RATCHET #4): после `npm run build` собирает internal href из `dist/**/*.html`, проверяет резолв в `dist/<path>/index.html`; для страниц `/ru/`(`/uk/`) — КАЖДАЯ внутренняя ссылка обязана нести свой префикс (иначе FAIL, кроме `.links-baseline.json`); ссылки переключателя языка (`<a hreflang>`) и hreflang-alternate в `<head>` не считаются нарушением. Требует существующий `dist/` (иначе exit 1 с подсказкой запустить build).

**Как НЕ допустить.** Привязка: `check-links` (#4). Для нового en-only сайта правило префикса неактивно (нет `/ru/`-дерева), но резолв внутренних ссылок и `/go/`-проверка работают. Прогонять `npm run build && npm run test:links` перед публикацией. Формат всегда `/{category}/{slug}/` со слешем.

## C7 — Статьи-сироты без входящих ссылок (<2 внутренних)

**Симптом.** Опубликованная статья не имеет входящих внутренних ссылок из тела других статей → «висячий лист», который не получает internal PageRank и плохо обнаруживается. Симметрично — статья без ≥2 исходящих контекстных ссылок.

**Как детектить.** Гейт `scripts/check-interlinks.mjs`: считает внутренние ссылки в ТЕЛЕ статьи/маршрута (markdown `](/...)` + raw `<a href="/...">`, исключая `/go/`, якоря, внешние; шаблонные крошки/RelatedPosts/nav не в счёт). Порог MIN=2. По умолчанию WARN (не валит), `--strict` → exit 1. De-orphan-проход (входящие ссылки на evergreen-статьи) уже сделан во всех 5 (коммиты `content(...): de-orphan evergreen articles`).

**Как НЕ допустить.** Привязка: `check-interlinks` (ratcheted в #2/#3/#4 серии) + STANDARDS «Перелинковка»: внутренняя ≥2 контекстных + анти-сироты (у каждой новой статьи должна появиться хотя бы 1 входящая из релевантной существующей) + кросс-сайтовые правила (контекстно, авто-nofollow, без PBN-ферм). Для нового сайта при публикации статьи N сразу добавлять входящую ссылку из тематически близкой статьи.

## C8 — future-date `publishedAt` (плановая дата как дата публикации)

**Симптом.** Статью публикуют (draft:false) сегодня, но в `publishedAt` стоит будущая плановая дата из календаря → статья «из будущего» (сортировка/sitemap `lastmod`/лента новостей врут; дашборд показывает опубликованное на будущей дате). Чинено в Албании дважды (`1b8267f`: Saranda/Berat были 01/10.07 при реальной 22.06; `48af398`).

**Как детектить.**
```
git -C <repo> grep -rhoE "^publishedAt:\s*'?\"?[0-9-]+" -- src/content | sort  # сверить с сегодня
```
Dashboard-парсер `grafana/build-status.mjs` несёт «future-date смелл» (строка ~140): опубликованная статья без явного `ОПУБЛИКОВАНО YYYY-MM-DD` → дата может быть плановым слотом. Для будущих ячеек дашборд показывает только «к написанию» (`○`), опубликованное на будущих датах не висит.

**Как НЕ допустить.** Правило /work + память `dashboard-sync-after-calendar-change`: `publishedAt` = **реальная дата публикации**, не плановый слот; даты новостей — только из источника текущего года (CLAUDE правило 5). На будущих датах в плане — только `○`. Для нового сайта то же: автопубликатор должен ставить `publishedAt = today` в момент `draft:false`, а не копировать дату из календаря.

## C9 — Рассинхрон дашборда / календарей / доков с реальностью

**Симптом.** После публикации/переноса слота KALENDAR.md, hub-копия `docs/calendars/*`, PROGRESS/HANDOFF/ROADMAP или дашборд расходятся с реальным состоянием репо. Парсер календаря «молча» терял слоты (дата не в начале строки; «Замер»/статья в одной строке; будущее показывалось как опубликованное).

**Как детектить.** Хардненный парсер (`grafana/build-status.mjs`, #5, коммит `c601e57`): диагностика молча выпадающих слотов — `looksTask`/`looksSlot` распознаёт строку со статусом/днём недели, но без даты → раньше выпадала. Память `calendar-read-omitted-long-lines`: «[Omitted long matching line]» в grep = матч; считать задачи на день счётчиком и сверять число. Память `dashboard-calendar-parser-pitfalls`: не матчить `includes(дата)`; статья > замер; будущее = только `○`.

**Как НЕ допустить.** Привязка: post-commit hook + 15-мин таймер пересобирают `grafana/build-status.mjs` + `build-html.mjs` после коммита; НО окно дашборда (вчера/сегодня/завтра/послезавтра) сверять глазами (память `dashboard-sync-after-calendar-change`).
