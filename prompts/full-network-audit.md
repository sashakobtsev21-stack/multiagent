# Промпт: полный аудит сети (хаб + 5 сайтов) под трафик и доход

> Запускать из **хаба** (`multiagent`). Это read-only аудит-оркестратор поверх per-repo `/full-audit`.
> Вставь этот файл целиком в чат Claude в корне хаба. Аудит НЕ правит код и НЕ публикует — только собирает проверенную картину и приоритизированный план. Фиксы — отдельной задачей после твоего «ок».

---

## 0. Роль и цель

Ты — **аудитор-дирижёр сети из 6 репозиториев**: хаб `multiagent` + 5 сайтов в `sites/` (`gruzia-site`, `albania-site`, `montenegro-site`, `croatia-site`, `macedonia-site`). Все 5 — форки одного движка (Astro 6.4.6 SSG, Tailwind 4, Leaflet, деплой Cloudflare Workers Static Assets, монетизация через `/go/`, GA4).

**Бизнес-цель сети — максимум органического трафика из Google и максимум дохода с рекламы/партнёрок.** Каждую находку оценивай через эту призму: «как это влияет на трафик и/или на деньги». Технически красивое, но не влияющее на цель — низкий приоритет; скучное, но режущее трафик/доход — высокий.

**Режим:** аудит read-only. Ничего не публиковать, не запускать `/work`/`/content`/`/news`, не снимать `draft`, не пушить фиксы. Любая правка — только после явного «ок» владельца, отдельным заходом.

**Язык отчётов:** русский.

## 2. Принципы аудита (соблюдать все)

1. **Проверяй реальность, а не доки.** Любое утверждение из README/PROGRESS/реестра подтверждай файлами. Расхождение «док ↔ факт» — само по себе находка.
2. **Severity:** `Critical` (ломает индексацию/деньги/сборку/безопасность) · `High` · `Medium` · `Low`.
3. **Теги влияния** к каждой находке: `💰доход` · `📈трафик/SEO` · `🛡безопасность` · `♿a11y` · `⚡перф` · `🔧поддерживаемость`.
4. **Каждая находка = факт + где + доказательство + влияние + причина + предлагаемый фикс + оценка усилий + вердикт «делать / не делать / отложить».** Цель владельца — понять, что чинить, на что это влияет и **нужно ли оно вообще**.
5. **Не ломать языковую политику:** Грузия en/ru/uk; Черногория/Хорватия en+ru; Албания/Македония только en. «Недостающие» ru/uk у одноязычных — НЕ баг.
6. **Не выдумывать** URL, цифры объёма поиска, факты. Нет данных — пиши «нет данных / проверить вручную».
7. **Файловые инструменты** (Glob/Grep/Read) — основной способ. Тяжёлые проверки (Lighthouse, реальная индексация, DNS) — где окружение не позволяет, явно помечай «прогнать вручную» и заноси в план.
8. **Не запускать `/work` и публикацию.** `npm run build`/`qa` для аудита — можно (сборка локальна, в прод не идёт).
9. **Кросс-сетевой взгляд:** ищи проблемы, общие для всех 5 форков (правишь один раз в движке-доноре → тиражируешь), и проблемы конкретного сайта.

## 3. Порядок (почему так)

Каждый этап готовит следующий, чтобы выводы не пришлось переделывать:

`A` Реальность и индексация → `B` Тех/код/зависимости → `C` Безопасность → `D` Тех-SEO → `E` SEO-ключи и календарь-темы → `F` Качество контента и фото → `G` Календари и автоматизация `/work` → `H` Монетизация vs цель → `I` UX/дизайн/перф/a11y → `J` Документация и чистка → `K` Консистентность хаба → синтез.

Сначала — **A (реальность/индексация)**: если домен сайта не привязан и Google его не видит, весь контент и SEO бессмысленны — это №1 по влиянию на цель.

## A. Реальность сети и индексация — 📈💰 КРИТИЧНО ПЕРВЫМ

Цель: понять, **видит ли вообще Google каждый сайт**. Без этого трафика ноль.

- **Домены/DNS/деплой.** По `docs/SITES-REGISTRY.md` домены 4 из 5 — «плейсхолдеры». Для каждого сайта определи: привязан ли кастомный домен к Cloudflare, резолвится ли, отдаёт ли 200, или сайт живёт только на `*.workers.dev`. (Где нет доступа — пометь «проверить: DNS/Cloudflare custom domain» и дай чек-лист владельцу.) **Сайт без живого домена = 0 индексации = высший приоритет.**
- **Google Search Console / Bing Webmaster.** Проверь наличие верификации (meta `google-site-verification` в `BaseLayout` или DNS-TXT). По находкам — GSC ни у кого не подтверждён в HTML. Без GSC нет данных о запросах (а календарь обещает «замеры GSC по понедельникам» — проверь, есть ли вообще откуда их брать).
- **Sitemap submitted.** `sitemap-index.xml` генерится — но подан ли в GSC? Корректен ли `lastmod`, не попадают ли `draft`/`/go/`/demo в sitemap (проверь filter в `astro.config`).
- **robots.txt / noindex.** Подтверди, что прод не закрыт случайно (`Disallow: /` нигде), а служебное (`/go/`, demo) закрыто. Проверь, нет ли `noindex` на боевых шаблонах.
- **canonical = реальному прод-URL** (с правильным доменом и trailing slash). На плейсхолдер-доменах canonical может указывать на несуществующий домен — это ломает индексацию.
- **Индексация по факту:** где возможно — оцени, сколько страниц реально в индексе (`site:домен`), есть ли санкции/ручные меры. Где нет доступа — чек-лист владельцу.

**Выход блока A:** таблица «сайт → домен живой? → GSC? → sitemap подан? → индексируется?» с вердиктом по каждому. Это фундамент приоритезации всего отчёта.

## B. Тех-аудит, код, зависимости (per-repo + сеть)

Для **каждого** сайта запусти его `/full-audit` (он мультиагентный, read-only, пишет `AUDIT-FULL-<дата>.md`) ИЛИ воспроизведи его логику. Плюс сверх него:

- **Гейты:** `npm run check` (типы), `npm run lint`, `npm run build`, `npm test` (enums+parity+photos+interlinks), `npm run test:links`, `npm run qa` (GO/NO-GO + аудит `dist/`). **Важно: `npm test` НЕ включает `test:links`** — проверь, ловятся ли битые ссылки штатным гейтом или только отдельной командой; если `/work` гоняет только `qa`, убедись, что `qa.mjs` реально вызывает links (в эталоне — да, фаза 1).
- **Зависимости/CVE.** `npm audit`. Зафиксируй: в build-tooling (esbuild/vite/astro) есть high/critical advisories — в задеплоенную статику не попадают, но реши: апгрейд мажора Astro нужен или нет (вердикт «делать/не делать» с обоснованием). Проверь lock-файлы, расхождение версий между форками (дрейф движка).
- **Дрейф форков.** Сравни версии и ключевые скрипты `scripts/*.mjs` между 5 сайтами — разъехались ли (правки в одном не доехали до других). Перечисли расхождения.
- **`dist/` в репозитории.** Проверь, не закоммичен ли собранный `dist/` (замечен в `croatia-site`) — это раздувает репо, даёт устаревшие артефакты и конфликты. Проверь `.gitignore` каждого.
- **Junk/секреты.** `*.stackdump`, `.DS_Store`, лишние `_scratch`, дубли `.gitkeep`. Секреты: не в текущих файлах И **в истории git** (`.env`, токены). CI-токены (`CLOUDFLARE_API_TOKEN` в `daily-news-rebuild.yml`) — только из secrets, не в коде.
- **CI/CD.** У каждого ли есть `.github/workflows/ci.yml` (check+build) и `daily-news-rebuild.yml`? Зелёные ли, одинаковые ли, не сломан ли cron новостей (он держит ленту 10/2 дня).

## C. Безопасность 🛡

- **CSP/`_headers`:** есть, без `unsafe-inline` в `script-src` (gtag — через sha256/nonce), HSTS, заголовки кэша. Сверь между сайтами — полнота одинаковая?
- **`/go/` open-redirect:** `worker/index.ts` редиректит ТОЛЬКО на цели из `partners.json`, неизвестный partner → безопасный fallback. Проверь во всех 5.
- **GDPR/Consent для EU-трафика.** 4 сайта таргетят Tier-1 Запад (EU/UK). Если планируется AdSense/медиасеть или уже стоит GA4 — нужен корректный **consent-баннер (Consent Mode v2)**. Проверь: есть ли он (в Хорватии замечен cookie-баннер + Consent Mode — есть ли у остальных?), не льёт ли GA4 данные до согласия, легально ли это для EU. Это и юр-риск, и условие монетизации рекламой в EU.
- **Лицензии фото.** Каждое фото — своё или CC/PD с атрибуцией (автор+лицензия+sourceUrl). Проверь выборочно, нет ли фото без атрибуции/с нарушением лицензии — это юридический и репутационный риск при росте трафика.

## D. Тех-SEO в масштабе 📈

- **Дубли slug / коллизии URL.** Подтверждённый баг: в `gruzia-site` `borjomi.md` (`category: dostoprimechatelnosti`) и `borjomi-gorod.md` (`category: goroda`) — **оба `slug: 'borjomi'`** во всех 3 языках → конфликт URL/canonical/sitemap. Проверь такие коллизии во всех репо (Grep по дублям `slug:`).
- **hreflang взаимность** между языковыми версиями (en↔ru↔uk): каждая версия ссылается на остальные + на себя + `x-default`. Цели hreflang существуют в `dist`. (В `qa.mjs` это есть для одного сайта — проверь во всех и кросс-язык.)
- **Битые внутренние ссылки и сироты, в т.ч. межрепные следы.** Подтверждённый пример: в `croatia-site` `InsuranceHub` ссылается на грузинский slug `kak-vybrat-strahovku`, которого в Хорватии нет → битая ссылка/сирота-хаб. Прогони `test:links` + Grep по хардкод-слугам донора во всех хабах/лейаутах.
- **trailing slash / редиректы.** `trailingSlash: always` — единообразно? Нет ли цепочек редиректов и петель в `_redirects`? URL после индексации не должны меняться без 301.
- **Структурированные данные → rich results.** JSON-LD есть (Article/Breadcrumb/Organization/ItemList/Trip). Проверь валидность (`qa.mjs` парсит) и **eligibility под rich snippets** (Breadcrumbs, при возможности FAQ/HowTo там, где уместно — сейчас FAQPage намеренно убран, оцени, не теряем ли расширенные сниппеты/AI Overview-видимость).
- **meta:** title ≤60 (без суффикса бренда), description ≤155, OG/Twitter — шаблоны во всех сайтах.
- **404 / RSS / пагинация.** 404 корректен; RSS (`/rss.xml`) не ссылается на несуществующие языковые ленты (в Македонии замечены упоминания `/ru/rss.xml`, `/uk/rss.xml` в `rss.ts` при en-only); пагинация листингов индексируема.
- **Каннибализация на уровне сети.** 5 сайтов = разные домены, ок. Но проверь дубли интента внутри одного сайта (см. блок E — календарь сам помечает «satellite/fold»).

## E. SEO-ключи и темы календаря — «темы из ТОП-запросов Google» 📈💰

Владелец требует: темы должны браться из **топ-запросов пользователей**. Проверь это строго.

- **Методология `docs/seo/*-seo-2026-06-22.md`.** Сейчас это качественный разбор кластеров (спрос/пробел конкурентов/интент/💰-связка), но **без числовых объёмов и сложности**. Оцени: достаточно ли этого? Для Грузии спрос русскоязычный (Wordstat/Яндекс), для остальных — английский (Google Keyword Planner). Зафиксируй: подтверждены ли темы реальными данными об объёме/конкуренции, или это экспертная оценка. Дай вердикт, нужно ли догружать количественные данные (Wordstat/KW Planner/Ahrefs/GSC).
- **Свежесть.** Отчёты от 2026-06-22, сайты живут с 24.06. Появились ли данные GSC/GA за период? Если да — сверены ли темы календаря с **реальными** растущими запросами (а не только с прогнозом)? Если GSC не подключён (см. блок A) — это разрывает заявленный цикл «корректируем план по замерам».
- **Соответствие интенту SERP.** Для топ-денежных тем (страховка/виза/аренда авто/ски-пасс/бюджет) проверь, что выбранный формат (гайд/лендинг/сравнение) бьётся с тем, что реально ранжируется в выдаче.
- **Каннибализация внутри плана.** Календарь Грузии сам пестрит пометками «🛰 САТЕЛЛИТ / fold / депри» — это риск двух статей на один интент. Проверь по всем 5 календарям: нет ли запланированных дублей родитель↔сателлит без `canonical`; соблюдается ли правило «не плодить вторую статью на тот же интент».
- **Покрытие денежных кластеров.** Сверь, что самые доходные кластеры (по `## Топ доходных тем` в SEO-доках) стоят в календаре первыми и усилены перелинковкой.
- **AI Overviews / SGE.** Оцени устойчивость тем к тому, что Google отвечает сам (where уместен экспертный угол, свежие 2026-данные, личный опыт — то, что AI Overview не закрывает).
- **Покрытие vs пробелы.** Сопоставь темы календаря с уже опубликованным `src/content/**` (не дублируем существующее) и с топ-кластерами SEO-дока (не пропускаем денежное).

## F. Качество контента и фото

- **Объём:** статья 1200–2000 слов, новость 1500–2000 знаков (3–5 разделов, ≥4 внутр. ссылки). Выборочно замерь, нет ли «тонких» статей.
- **EEAT / анти-ИИ.** Текст должен читаться как у живого носителя (en — native; ru/uk — живая речь, не калька), без «генерёнки», с уникальными фактами/опытом. Выборочно оцени тон и пробей на типичные ИИ-паттерны. Это критично и для ранжирования, и для доверия (а значит конверсии в деньги).
- **Достоверность/YMYL.** Факты только из источника; **даты — текущего года** (был реальный прокол с прошлогодними датами фестиваля); визы/налоги/ВНЖ/страховка — официальный источник + видимый дисклеймер. Проверь `<!-- TODO -->`-маркеры в контенте (в Албании замечены TODO по ценам/часам).
- **Фото-стандарт:** статья ≥5 (cover+≥4), новость ≥2, маршрут — фото каждой остановки, город ≥10, столица ≥15; webp ≤200 КБ; уникальные; атрибуция. **Подтверждённое нарушение:** в `albania-site` 5 из 16 статей — cover-only (0 инлайн); рестораны во всех репо часто без инлайн-фото. Прогони `check-photos` и сведи дефицит по каждому сайту.
- **Дубли методологии.** Один и тот же абзац-болванка в разных статьях — антипаттерн, проверь.
- **Контент-decay.** Старые статьи с устаревшими ценами/правилами — кандидаты на обновление (важно для удержания позиций).

## G. Календари и автоматизация `/work` 🔧📈

Владелец каждый день вводит `/work` → система по `KALENDAR.md` делает контент/новости. Надёжность этого цикла = надёжность роста.

- **Формат `KALENDAR.md` во всех 5.** У Грузии — детальный по дням (21 неделя). Проверь, что у остальных 4 календари того же формата, покрывают период, темы из SEO-доков, реалистичный темп, проставлены дни новостей и `📊 Замер GA/GSC`.
- **Hub-копии vs репо.** `docs/calendars/*-KALENDAR.md` (хаб) синхронны с `sites/<repo>/KALENDAR.md`? Кто источник правды? Расхождения — находка.
- **`/work` end-to-end dry-run.** Прогони логику `/work` в режиме разбора (без публикации): на тестовую дату — корректно ли находит `[Статья]`/`📰 /news`/`📊 Замер`, строит матрицу, понимает языки сайта, гейт согласования. Найди места, где может упасть/задвоить/опубликовать без «ок».
- **Git-надёжность.** `/work` зависит от `git pull --ff-only` и `git push` по 6 репозиториям. Проверь: чистые ли рабочие деревья, не разошлись ли с `origin/main`, нет ли незапушенного, нет ли конфликтов/завязших credential-хелперов (если `git` где-то зависает — это блокер всего цикла, занеси Critical). Подтверди, что `draft:true` реально исключается из сборки (Astro), а публикация = `draft:false` + push.
- **Daily-news-rebuild.** Cron `daily-news-rebuild.yml` жив, держит окна `NEWS_SECTION_WINDOW=10` / `NEWS_HOME_WINDOW=2`. Проверь во всех 5.
- **Гейт согласования.** Подтверди, что `/work`/`/news`/`/content` физически останавливаются до «ок» и не автопостят.
- **Локализация `/news` по странам.** Подтверждено: Грузия/Черногория/Хорватия/Македония — локализованы; в `albania-site` шаги `/news` ещё упоминают «тройку en/ru/uk» (рассинхрон с en-only). Проверь источники/валюту/гео в `.claude/skills/news/SKILL.md` каждого.

## H. Монетизация vs цель «заработок на рекламе» 💰

> ⚠️ **Стратегическая развилка.** Цель владельца — «зарабатывать на рекламе». Но сеть сейчас монетизируется **только партнёрками** (`/go/` + AffiliateBox); **дисплейная реклама (AdSense/Mediavine/Ezoic) нигде не подключена** — в Грузии она осознанно отклонена (`ROADMAP-GROWTH-2026.md`: конфликт CSP/перф, near-zero RPM на ru/uk). Это надо явно вынести владельцу как решение.

- **Пересмотреть display ads под цель.** Оцени по каждому гео: где дисплей-реклама даст доход (EU/UK Tier-1 трафик Албании/Черногории/Хорватии/Македонии — высокий RPM в EUR), а где почти ноль (ru/uk Грузии). Дай рекомендацию: подключать ли AdSense/медиасеть, на каких сайтах, с какими порогами трафика (Mediavine/Ezoic/Raptive имеют пороги по сессиям), как это уживётся с текущим строгим CSP и перф-бюджетом (JS ≤50 КБ/стр) и consent-баннером EU. Вердикт «делать/не делать/когда».
- **Affiliate-гигиена.** `/go/{partner}` + `rel="sponsored nofollow noopener"` — везде. **EKTA сломана во всех репо** (`allowSubId:false`, прямой URL без subid → атрибуция/комиссия теряется) — посчитай влияние и приоритет. Проверь все партнёрские ссылки на корректность шаблонов и subid.
- **Концентрация риска.** В Грузии ~84% через Trip.com — единая точка отказа дохода. Оцени диверсификацию по всем сайтам.
- **Готовность к деньгам = трафик.** Честно свяжи: партнёрки и реклама дают доход только при трафике, а трафик упирается в блок A (домены/индексация). Выстрой причинно-следственную цепочку приоритетов «сначала индексация → потом контент/SEO → потом монетизация».
- **Ненавязчивость и Page Experience.** Реклама не должна убивать CWV (иначе просадка позиций → меньше трафика → меньше денег). Учти это в рекомендации.
