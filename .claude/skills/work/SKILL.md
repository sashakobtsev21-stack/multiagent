# /work — утренний оркестратор по 5 сайтам на сегодня

Одна команда утром → готовые статьи + новости для всех сайтов, у кого они сегодня по плану → **один** показ владельцу → «ок/не ок» → публикация только одобренного.

**Команда:** `/work [страны] [дата]` — обе опц. По умолчанию все 5 сайтов на СЕГОДНЯ. Можно `/work грузия` или `/work все 24.06`.

## Карта сайтов
| Страна | Клон | Языки публикации | Каденс новостей |
|---|---|---|---|
| Грузия | `sites/gruzia-site` | en/ru/uk | Пн + Чт (+Сб пик) |
| Албания | `sites/albania-site` | en | Вт + Сб (+Ср Jul–Aug) |
| Черногория | `sites/montenegro-site` | en + ru | Пн + Ср + Пт |
| Хорватия | `sites/croatia-site` | en + ru | Пн + Чт (+Сб пик ~10–20.07) |
| Македония | `sites/macedonia-site` | en | Вт (+событийно/пик) |

## Процедура (гейт согласования НЕ пропускать)

### 1. Дата и разбор плана
- Определи сегодняшнюю дату (из окружения; если в `/work` передана дата — взять её).
- Для каждого целевого сайта: `git -C sites/<repo> pull --ff-only`, открой `sites/<repo>/KALENDAR.md`.
- Найди на сегодняшнюю дату:
  - **`[Статья]`** (ещё НЕ `[x]` опубликовано) → задача **content**. Возьми из строки: заголовок, целевой запрос, slug, категорию, мин. фото, 💰-связку, перелинковку.
  - **`📰 /news`** (день новостей вписан построчно; или вычисли по каденсу сайта на этот день недели) → задача **news**.
  - **`📊 Замер GA/GSC`** → не контент: напомни владельцу прислать данные GA (план корректируется по ним), статью в этот день не требуем.
  - Если на сегодня пусто или всё `[x]` → сайт пропускаем.
- Собери матрицу `сайт × {статья?, новости?}` и кратко покажи владельцу, что берётся в работу (1 строка на сайт) — затем сразу к производству (это не гейт публикации, просто прозрачность).

### 2. Производство ГОТОВЫХ черновиков (параллельно, ОДИН писатель на клон)
По каждому сайту с задачами — запусти агента в его клоне, который выполняет ПОЛНЫЙ конвейер, но оставляет `draft:true` (НЕ снимает draft, НЕ публикует live):
- **Статья** — по `.claude/skills/content/SKILL.md`: бриф → `npm run new` → 1200–2000 слов живого текста (лид=прямой ответ, без FAQ) → фото по стандарту (статья ≥5 / город ≥10 / столица ≥15, webp ≤200КБ, CC/своё с атрибуцией + sourceUrl на новых) → переводы на языки сайта (паритет slug/числа) → fact-checker → content-editor → `npm run qa` зелёный. Оставить `draft:true`.
- **Новости** — по `.claude/skills/news/SKILL.md`: сбор по курируемым источникам сайта → 3-ступ. фильтр → **cross-site-fact-checker (дата/место/год/реальность, ловить прошлогодние даты)** → рерайт 1500–2000 знаков, 3–5 разделов, ≥4 внутр. ссылки → фото ≥2 (cover ≤200КБ + ≥1 инлайн) → переводы → `qa`. Оставить `draft:true`. Если из фильтра ничего не прошло — пометить «новостей на сегодня нет (skip)».
- Один пишущий агент на клон (память `one-writer-per-clone`); если у сайта и статья, и новости — один агент делает обе ПОСЛЕДОВАТЕЛЬНО. Параллелить только между разными сайтами.
- Черновики коммитить как `draft:true` (забанковать, в прод не попадают — Astro исключает draft из сборки) ИЛИ держать локально; на обрыв сессии — забанкованное не теряется.

### 3. ОДИН пакет на утверждение — СТОП
Сведи всё в одно сообщение, сгруппированное по сайтам, показав РЕАЛЬНЫЙ готовый контент (не план):
```
## 🇬🇪 Грузия
### 📄 Статья: <заголовок>  (/slug/, категория)
объём N слов · M фото (автор/лицензия) · перелинк ✓ · qa=GO
лид-абзац: «…»  +  список H2-разделов
### 📰 Новости (K шт.)
| # | заголовок | суть 1–2 строки | источник | фактчек ✓ |
## 🇦🇱 Албания
…
```
По каждому: объём, фото, qa-статус, перелинк. **Ничего не идёт в прод до явного «ок».** Владелец может одобрить ВЫБОРОЧНО: «ок всё», «ок Грузия+Хорватия», «новости ок, статью Албании переписать» и т.п.

### 4. После «ок» — публикация ТОЛЬКО одобренного
По каждому одобренному материалу в его репо:
- `draft:false` → `npm run qa` = **GO** → один коммит → `git pull --rebase origin main` → `git push origin main` (= деплой).
- Партнёрки только `/go/`. Языковой паритет соблюдён.
- Отметь опубликованное в `sites/<repo>/KALENDAR.md` (`[x]` + slug/дата) и синхронизируй hub-копию `docs/calendars/<short>-KALENDAR.md`; обнови `PROGRESS.md`/`HANDOFF.md`, закоммить в репо сайта.
- **Хаб тоже под git:** после правок hub-копий календарей (`docs/calendars/*`) и любых изменений скиллов/доков хаба — `git -C . add` → `git commit` → `git push` в репо хаба `sashakobtsev21-stack/multiagent` (ветка `main`), тем же заходом. Не оставлять hub-копии только на диске.
- **«Не ок»**: не публикуй. Если просили правки — внеси, покажи снова (вернись к п.3 по этой позиции). Если отклонено совсем — оставь `draft:true` или удали.

### 5. Отчёт владельцу
- Что и куда опубликовано (по сайтам), что пропущено/отклонено, что осталось черновиком. Напомни про дни `📊 Замер GA/GSC`, если были сегодня.
- **ОБЯЗАТЕЛЬНО — ссылки на новый контент в конце сообщения (только ПОСЛЕ полного `git push`).** Когда весь одобренный материал реально запушен (= задеплоен), последним блоком отчёта выведи в чат **кликабельные полные URL на каждый созданный сегодня материал — и статьи, И новости**, по всем языковым версиям сайта. Это финальный блок, чтобы владелец одним кликом открыл каждый новый материал.
  - **URL строй** из продакшн-домена сайта (`docs/SITES-REGISTRY.md`) + языковой префикс (en — корень `/`; ru → `/ru/`; uk → `/uk/`) + `{категория}/{slug}/` со слешем. Пример (Грузия, домен `georgiaguidebook.com`): `https://georgiaguidebook.com/<cat>/<slug>/`, `https://georgiaguidebook.com/ru/<cat>/<slug>/`, `https://georgiaguidebook.com/uk/<cat>/<slug>/`.
  - Формат — markdown-ссылки, сгруппированы по сайтам, с пометкой типа (📄 статья / 📰 новость).
  - Если у сайта домен в реестре помечен **плейсхолдером** и кастомный домен не привязан к Cloudflare — выведи URL по домену из реестра, но припиши, что нужно проверить привязку домена (иначе материал доступен только на `*.workers.dev`).
  - Выводи ссылки **только на то, что прошло push** (не на черновики и не на отклонённое). Если push по какой-то позиции не прошёл — не давай ссылку, явно скажи, что не задеплоено.

### 6. Конец недели — календарь на СЛЕДУЮЩУЮ неделю (ОБЯЗАТЕЛЬНО)
В ПОСЛЕДНИЙ рабочий день недели (обычно Пт, или последний день недели, на который были задачи) — по завершении публикации выведи владельцу **календарь на следующую календарную неделю (Пн–Вс) по ВСЕМ 5 сайтам**, день-в-день:
- Таблица: строки = даты следующей недели, столбцы = 5 сайтов; в ячейках — что стоит (статья/маршрут, 📰 новости, 📊 замер). Легенда: ✅ опубл · ○ к написанию · ⭐ SEO-голова · 📰 · 📊.
- Данные бери из `sites/<repo>/KALENDAR.md` каждого сайта на даты следующей недели.
- Цель: владелец видит весь предстоящий объём до старта следующего цикла `/work`.

## Запреты (наследуются от /news, /content, памяти)
- Ничего в прод без явного «ок». Факты только из источника (даты — текущий год; для ежегодных событий не брать прошлогодние). Фото обязательны, с атрибуцией (новые — с sourceUrl на Commons). Партнёрки только через `/go/{partner}`. YMYL (визы/налоги/ВНЖ/страховка) — официальные источники + видимый дисклеймер. Языки строго по сайтам. Один писатель на клон. `npm run qa` зелёный перед публикацией. URL после индексации не менять (301 в `_redirects`).
- Не публиковать половину языковой пары/тройки; не копировать чужой текст дословно; не выдумывать цены/часы (→ visit + checkedAt + «уточняйте» или TODO).
