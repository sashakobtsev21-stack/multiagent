# /work — утренний оркестратор по 5 сайтам на сегодня

Одна команда утром → выполняю **весь цикл от начала до конца** по всем сайтам, у кого сегодня есть план → по каждому материалу **жёсткая самопроверка** → **при полной уверенности публикую сам в прод** (деплой). Останавливаюсь и спрашиваю владельца **только если возникло уточнение** (неясный/непроверяемый факт · спорный YMYL · неоднозначный таргет ссылки · сомнение в качестве). Где всё чисто — не жду «ок».

**Команда:** `/work [страны] [дата]` — обе опц. По умолчанию все 5 сайтов на СЕГОДНЯ. Можно `/work грузия` или `/work все 24.06`.

### Новая логика контента (с 2026-06-28, согласовано владельцем)
- **Новости — 1 дайджест/нед, по субботам** (Сб — свободный от статей день, не конкурирует с темпом будней). Механика прежняя: сбор → 3-ступ. фильтр (тревел-релевантное **текущего 2026**) → cross-site-fact-checker → превью владельцу → «ок» → публикация. **Пустая суббота — норма** (skip, если фильтр пуст). Исключение: реально срочная новость (новый рейс/виза/закрытие) может выйти внеочередным мини-дайджестом, не дожидаясь субботы.
  - **Что входит в новостной день (не только «жёсткие» новости, решение владельца 2026-06-30):** (а) критичные инфра/визовые/транспортные изменения (рейсы, визы, границы, цены, ЧС/погода, открытия-закрытия); **(б) тревел-значимые СОБЫТИЯ** — фестивали, городские праздники, сезонные и культурные события, ярмарки, концерты, спортивные/гастрономические события, юбилеи — то, что полезно знать путешественнику, планирующему поездку в эти даты. Главное — реальное, датированное **текущим годом**, со ссылкой на источник (🔴 для ежегодных событий не брать прошлогодние даты — память [[network-content-lessons]]).
  - **Количество — по факту:** сколько реально набралось стоящего (ориентир 3–5 на сайт, но без добивки ради числа). Мало стоящего → меньше; пусто → пустая суббота. Событие можно дать и отдельной новостью, и собрать в мини-обзор «что посмотреть в этом месяце / афиша».
- **Thin-темы НЕ занимают будние слоты.** Низкочастотные информационные микро-темы без коммерческого оффера (розетки/переходники, вода из-под крана, мелочи о валюте и т.п.) НЕ публикуем отдельными статьями — вплетаем разделом в родительский хаб (Safety / Budget / Packing / General Guide). (FAQ-блоков нет — вплетаем в тело прозой.) Это снижает риск thin-content и экономит слоты.
- **Освободившиеся будние слоты → коммерция (приоритет 💰):** (а) транспортные связки «Точка А → Точка Б» (bus/car/taxi/ferry) с трансфер/авто-партнёрками; (б) «Where to stay в <локация/пляж>» с отельными офферами; (в) сравнения локаций «Курорт_1 vs Курорт_2». Это decision-stage, высокая конверсия — заполнять ими в первую очередь.
- **Стиль:** живой travel-эксперт «с места» — реальные цены/расписания/координаты/локальные блюда **только из источника + дата + „уточняйте“** (не выдумывать — YMYL-риск). Опубликованное `[x]` неприкосновенно.
- **Темп новых (с 2026-06-30, решение владельца):** **3 статьи/будень (Пн–Пт) на каждый из 4 EN-сайтов** (Албания/Черногория/Хорватия/Македония) = 15/нед на сайт. **Грузия — 1/будень** (зрелый флагман, 91+ статья, осознанно медленнее; ru/uk/en = тройная работа на тему). Календари размечены на 3/будень; Сб/Вс — без статей (Сб = новости). См. память [[content-cadence-3-per-weekday]].

## Контекст и непрерывность (НЕ терять контекст в процессе)
Длинный прогон по 5 сайтам — банкуй прогресс инкрементально, не держи всё в голове:
- Веди **живой чек-лист задач** (сайт × {статья / новости}) и отмечай выполненное по ходу; если задач много — держи состояние в файле scratchpad, а не только в памяти.
- **Каждый готовый материал банкуй СРАЗУ:** прошёл самопроверку → публикуй и пушь; есть вопрос → коммить `draft:true`. Не накапливай несохранённое — на обрыв сессии запушенное/забанкованное не теряется.
- **После КАЖДОГО сайта обновляй `HANDOFF.md`** (снимок: что сделано, где остановился, что дальше) — это точка возобновления, если сессия прервётся.
- Идёшь сайт за сайтом ДО КОНЦА; не бросаешь на середине без отметки в HANDOFF. Один писатель на клон (память `one-writer-per-clone`); параллелить только между разными сайтами.
- Перед каждым push — `git pull --rebase origin main` (п.4): ловит любой чужой/параллельный коммит и не даёт затереть.

## Процедура (автономно от начала до конца; гейт — только на уточнениях)

### 1. Дата и разбор плана
- Определи сегодняшнюю дату (из окружения; если в `/work` передана дата — взять её).
- Для каждого целевого сайта: `git -C sites/<repo> pull --ff-only`, открой `sites/<repo>/KALENDAR.md`.
- Найди на сегодняшнюю дату:
  - **`[Статья]`** (ещё НЕ `[x]` опубликовано) → задача **content**. Возьми из строки: заголовок, целевой запрос, slug, категорию, мин. фото, 💰-связку, перелинковку.
  - **`📰 /news`** (день новостей вписан построчно; или вычисли по каденсу сайта на этот день недели) → задача **news**.
  - **`📊 Замер GA/GSC`** → не контент: напомни владельцу прислать данные GA (план корректируется по ним), статью в этот день не требуем.
  - Если на сегодня пусто или всё `[x]` → сайт пропускаем.
- Собери матрицу `сайт × {статья?, новости?}` и кратко покажи владельцу, что берётся в работу (1 строка на сайт) — затем сразу к производству (это не гейт публикации, просто прозрачность).

### 2. Производство до готовности (параллельно между сайтами, ОДИН писатель на клон)
По каждому сайту с задачами — выполни в его клоне ПОЛНЫЙ конвейер до готовности (`draft:true` — промежуточный банк; снятие draft и публикация — в п.3–4 после ревью). Конвейер статьи = **ресёрч-бриф → текст → фото → перевод** (порядок важен — бриф первым снижает выдумки и удешевляет фактчек):
1. **Ресёрч-бриф СНАЧАЛА (анти-выдумка).** У `/work` нет готового брифа владельца, поэтому факты (расписания/цены/расстояния/операторы) добывает агент (`research-collector` или сам писатель через web) и складывает в `briefs/{slug}.md` — **каждый твёрдый факт с URL-источником**, диапазоны + дата проверки. Только проверяемое; нет подтверждения → «уточняйте»/TODO, НЕ выдумывать (YMYL — официальные первоисточники). Бриф потом служит опорой `fact-checker` (сверка с брифом дешевле, чем ресёрч заново).
2. **Текст из брифа.** `npm run new` → 1200–2000 слов по `docs/STANDARDS.md`: лид=прямой ответ; **гейт ② человечность + интересность** (ноль ИИ-штампов; голос практика «кто реально ездил»; 1–2 не-очевидных инсайта; конкретные сравнения «что выбрать»; без FAQ). Партнёрки 1–3 только `/go/`.
3. **Фото по стандарту.** Статья ≥5 / город ≥10 / столица ≥15, webp ≤200КБ, CC/своё с атрибуцией + sourceUrl. **Уникальные кадры** (НЕ один файл в gallery+inline ради счётчика — реальный прокол), отбор глазами. Через `scripts/commons-*` напрямую, без вложенных под-агентов.
4. **Переводы на языки сайта** (en/ru/uk по карте) с **семантической верностью** (по смыслу, не дословно — урок «платки»=tolls), паритет slug/чисел/картинок.
   Оставить `draft:true`. **Входящие ссылки на новую страницу ставить ТОЛЬКО когда она уже `draft:false`** (draft не собирается → ссылка на него = «битая» в `test:links`); поэтому входящие добавляй когортой в момент публикации (п.4), не на этапе черновика.
- **Новости** — по `.claude/skills/news/SKILL.md`: сбор → 3-ступ. фильтр → **`cross-site-fact-checker` (дата/место/год/реальность, ловить прошлогодние даты)** → рерайт 1500–2000 знаков, 3–5 разделов, ≥4 внутр. ссылки → фото ≥2 → переводы → `draft:true`. Пусто из фильтра → «новостей на сегодня нет (skip)».
- Один пишущий агент на клон (память `one-writer-per-clone`); статья+новости одного сайта — один агент ПОСЛЕДОВАТЕЛЬНО. Параллелить только между разными сайтами. **Банкуй каждую готовую статью отдельным `draft:true`-коммитом + push СРАЗУ** (краш-безопасность, память `work-crash-safety`); на обрыв сессии забанкованное не теряется (Astro исключает draft из сборки — в прод не попадёт).

### 3. Ревью качества — РЕДКОЛЛЕГИЯ в слоях (независимо от писателя; с экономией токенов)
Каждый материал проходит гейт перед публикацией. **Главный принцип эффективности: скрипт там, где можно скриптом (0 токенов); агент — ТОЛЬКО на то, что скрипт не умеет (смысл, правдивость, живость, естественность языка).** Независимость обязательна: ревьюер ≠ контекст, что писал (память `human-tone-hard-gate`). Слои гони ПО ПОРЯДКУ — дешёвое отсекает до трат на дорогом.

**Слой 0 — детерминированные скрипты (0 токенов, ПЕРВЫМИ; красное → чинить ДО агентов):**
- `npm run qa` = GO — читать строку **`ВЕРДИКТ: GO` + «критических 0»**, НЕ exit code (память `qa-verdict-not-exit-code`): build + SEO-мета (title≤60/desc≤155 в СИМВОЛАХ, память `seo-meta-length-chars-not-bytes`) + hreflang + schema + canonical + alt + a11y + перф. В составе `qa`/`npm test`: `check-dedup` (④ анти-каннибализация), `check-parity` (паритет языков), `check-photos` (≥5 фото).
- `npm run test:links` = GO (0 битых; перелинковка в обе стороны; нет ссылок на draft/несуществующее).
- `npm run qa:responsive` — мобайл-overflow. При изменении ШАБЛОНА/визуала (не контента) — `npm run qa:browser` (Lighthouse ≥90) и/или агент `ui-ux-auditor` ОДИН раз на дизайн-изменение, **НЕ на каждую статью** (статьи на одном шаблоне → вёрстка одинакова).
→ Слой 0 закрывает даром: ссылки/перелинковку · SEO-мету · паритет en/ru/uk · фото-гейт · мобайл-вёрстку. Дальше агенты — только на смысл.

**Слой 1 — смысловые спец-ревьюеры на Sonnet (ПАЧКОЙ на сайт = 3 статьи разом, не по одной; независимый контекст):**
- 🔴 `fact-checker` (Sonnet, web) — **правдивость:** каждый твёрдый факт (цены/расписания/расстояния/даты) сверить с брифом/источником; даты текущего года (для ежегодных событий не брать прошлогодние); ноль выдумок; YMYL — официальные первоисточники.
- 🔴 `content-editor` (Sonnet) — **человечность + интересность + грамматика:** чек-лист ② `docs/STANDARDS.md` (лид-крючок; ноль ИИ-штампов; конкретика; живой ритм; голос практика; **интересно ли дочитать до оффера**). Скучно-но-корректно → на доработку.
- 🔴 **языковой ревью** (Sonnet) — **только сайты с ru/uk** (Грузия/Черногория/Хорватия): ru/uk звучат как родной (не калька) И **семантически верны** — ловит «платки»=tolls (грамматически верно, по смыслу мусор; скрипт это НЕ видит); паритет смысла с en. EN-only (Албания/Македония) — слой пропустить.

**Слой 2 — «ГЛАВНЫЙ РЕДАКТОР» на Opus, 1 на САЙТ (не на статью):** читает 3 готовых черновика + вердикты слоя 1 и выносит финальное **GO / NO-GO по сути:** правдиво? живо и интересно? нет ли того, что слои упустили? Это самый опытный независимый глаз — дорогой, поэтому ОДИН проход на сайт, на вход — уже отфильтрованное дешёвыми слоями.

**Правила экономии токенов (соблюдать всегда):**
1. Слой 0 — первым; не звать агента на то, что ловит скрипт.
2. Ревью **пачкой на сайт**, не по 1 статье (один вызов агента на 3 статьи).
3. После МЕХАНИЧЕСКОЙ правки (фото/мета/ссылка/паритет) — перегнать только нужный СКРИПТ, **НЕ перезапускать смысловое ревью** заново.
4. Визуал/UX/SEO-шаблон — на уровне ШАБЛОНА раз в дизайн-изменение, не на каждую статью.
5. Дорогой Opus — только «главред» (1/сайт) и единичный спорный факт; всё остальное Sonnet/скрипт.
6. Хороший ресёрч-бриф (п.2) → меньше переделок; переделка = двойной расход токенов (реальный прокол: целый перезапуск писателя из-за того, что ловилось дешёвой проверкой).

**Решение о публикации:** все слои зелёные и нет уточнений → **evergreen не-YMYL** статью публикуешь сам (п.4), не дожидаясь «ок». **YMYL** (визы/налоги/ВНЖ/резидентство/страховка) и **ВСЕ новости** — всегда owner-preview (`draft:true` + показать на «ок»), ДАЖЕ при зелёных гейтах (решение владельца 2026-06-30).

**Гейт уточнения (единственный стоп для evergreen):** любой пункт под вопросом — непроверяемый факт, спорный YMYL, неоднозначный/несуществующий таргет ссылки, сомнение в качестве/правдивости/интересности — **этот материал НЕ публикуй**, вынеси владельцу с конкретным вопросом и жди ответа. Остальные (где чисто) — публикуй сразу, не задерживай из-за одной спорной позиции.

Для прозрачности — короткая сводка (1 строка на сайт: что опубликовано / что отложено с вопросом) — отчёт по ходу, НЕ блокирующий показ.

### 4. Публикация (того, что прошло самопроверку)
По каждому материалу с зелёной самопроверкой, в его репо:
- `draft:false` → `npm run qa` = **GO** → `npm run test:links` = **GO** → один коммит → `git pull --rebase origin main` → `git push origin main` (= деплой).
- Партнёрки только `/go/`. Языковой паритет соблюдён.
- Отметь опубликованное в `sites/<repo>/KALENDAR.md` (`[x]` + slug + **реальная дата публикации**, не плановый слот) и синхронизируй hub-копию `docs/calendars/<short>-KALENDAR.md`; обнови `PROGRESS.md`/`HANDOFF.md` (снимок), закоммить в репо сайта.
- **Хаб тоже под git:** после правок hub-копий календарей (`docs/calendars/*`) — **пересобери дашборд** (`node grafana/build-status.mjs && node grafana/build-html.mjs`), затем `git -C . add` → `commit` → `push` в репо хаба `sashakobtsev21-stack/multiagent` (память `dashboard-sync-after-calendar-change`, `all-docs-always-current`). Не оставлять hub-копии/доки только на диске.
- **Позиция с вопросом (гейт уточнения, п.3):** оставь `draft:true` до ответа владельца; внёс правки по ответу — повтори самопроверку п.3 и публикуй. Отклонено совсем — удали или оставь черновиком.

### 5. Отчёт владельцу
- Что и куда опубликовано (по сайтам), что пропущено/отклонено, что осталось черновиком. Напомни про дни `📊 Замер GA/GSC`, если были сегодня.
- **ОБЯЗАТЕЛЬНО — ссылки на новый контент в конце сообщения (только ПОСЛЕ полного `git push`).** Когда весь одобренный материал реально запушен (= задеплоен), последним блоком отчёта выведи в чат **кликабельные полные URL на каждый созданный сегодня материал — и статьи, И новости**, по всем языковым версиям сайта. Это финальный блок, чтобы владелец одним кликом открыл каждый новый материал.
  - **URL строй** из продакшн-домена сайта (`docs/SITES-REGISTRY.md`) + языковой префикс (en — корень `/`; ru → `/ru/`; uk → `/uk/`) + `{категория}/{slug}/` со слешем. Пример (Грузия, домен `georgiaguidebook.com`): `https://georgiaguidebook.com/<cat>/<slug>/`, `https://georgiaguidebook.com/ru/<cat>/<slug>/`, `https://georgiaguidebook.com/uk/<cat>/<slug>/`.
  - Формат — markdown-ссылки, сгруппированы по сайтам, с пометкой типа (📄 статья / 📰 новость).
  - Если у сайта домен в реестре помечен **плейсхолдером** и кастомный домен не привязан к Cloudflare — выведи URL по домену из реестра, но припиши, что нужно проверить привязку домена (иначе материал доступен только на `*.workers.dev`).
  - Выводи ссылки **только на то, что прошло push** (не на черновики и не на отклонённое). Если push по какой-то позиции не прошёл — не давай ссылку, явно скажи, что не задеплоено.

## Типичные грабли — проверять ДО публикации (разбор прогона 29.06.2026)
Эти ошибки реально возникали и роняли `qa`/`build`/`test:links` или теряли работу. Профилактика — встроена в самопроверку п.3, но держи в голове отдельно:
1. **`qa`/`test:links` выходят с кодом 0 ДАЖЕ при NO-GO.** Проверять текст вердикта (`ВЕРДИКТ: GO` + `критических 0`), а НЕ exit code (память [[qa-verdict-not-exit-code]]). Чуть не опубликовал сломанные Хорватию/Черногорию по «exit 0».
2. **`title ≤60` / `description ≤155` — в СИМВОЛАХ (не байтах) — это ZOD-лимиты, роняющие `build`.** Превышение = `check`/`build` NO-GO. Проверять длину ДО `draft:false` (Хорватия: 6 desc>155 + 2 title>60 уронили build). Память [[seo-meta-length-chars-not-bytes]].
3. **ru/uk-версии: каждая внутр. ссылка с `/ru/`(`/uk/`)-префиксом**, не на en-корень; и **НЕ ссылаться на draft/несуществующие страницы** — взаимосвязанный набор публиковать вместе (Черногория: ru вёл на en-URL + на draft-маршрут → 8 битых). Память [[astro-internal-link-format]].
4. **Банковать КАЖДУЮ готовую статью отдельным коммитом + push СРАЗУ** (краш-безопасность): рестарт процесса убивает фоновых писателей, теряется незакоммиченное. И **закоммичено ≠ запушено** — после публикации сверить push (= деплой). Память [[work-crash-safety]].
5. **Фото — через `scripts/commons-*` напрямую**, не плодить вложенные под-агенты для подбора (застревают в цепочке). Один писатель на клон, без глубокой вложенности.
6. **Категория из KALENDAR может расходиться с реальной** (план писал `attractions`, файл — `category: transport`). Публикуй под РЕАЛЬНОЙ категорией статьи, в календаре отметь РЕАЛЬНЫЙ URL.
7. **Семантика перевода (прогон 01.07): `tolls` → «платки» (=шарфики) вместо «плата за проезд»** в 6 местах Хорватии (вкл. 2 уже опубликованные статьи). Грамматически верно — скрипт НЕ ловит; ловит только языковой ревьюер-носитель (слой 1). Частые ловушки: toll/fee/fare/deposit/excess/return/carrier. → языковой ревью обязателен для ru/uk перед публикацией.
8. **Порядок draft-ссылок (прогон 01.07): входящие ссылки на новую страницу — ТОЛЬКО когда она уже `draft:false`.** Draft не собирается → ссылка на него = «битая» в `test:links`. Нельзя одновременно «входящие есть» + «страница draft» + «test:links зелёный». Решение: флипнуть когорту в `draft:false`, ПОТОМ добавить входящие, ПОТОМ `qa`/`test:links` (тогда таргеты собираются и ссылки резолвятся).
9. **Счётчик фото обманывается дублем (прогон 01.07): один файл в `gallery` + он же inline `<figure>` = `check-photos` считает 5, а уникальных 4.** Нужны 5 РАЗНЫХ файлов. 5-й часто уже скачан в папку, но не вставлен — проверить уникальность по именам файлов, не по счётчику гейта.
10. **Сначала скрипт, потом агент (экономия токенов).** Прогоняй Слой 0 (qa/test:links/responsive) ДО смысловых агентов — механику (мета/ссылки/паритет/фото/overflow) ловит скрипт даром; не жги Opus/Sonnet на то, что поймает `npm run qa`. После механической правки — перегон только скрипта, не повторное смысловое ревью.
