# CLAUDE.md — правила работы в хабе

Это **хаб-дирижёр** над 5 сайтами-путеводителями. Общайся с владельцем по-русски.

## Что такое хаб и чего он НЕ делает
- Хаб оркестрирует, но **не дублирует** per-repo логику. Реальную работу (написание, фото, перевод, QA, публикация) выполняют скиллы и агенты ВНУТРИ каждого репо (`/news`, `/add-content`, `/full-audit`).
- Клоны репозиториев лежат в `sites/<repo>/`. Перед работой — `git pull`, после — `git commit` + `git push` в `main` (деплой = push).
- **Сам хаб — приватный git-репозиторий** `sashakobtsev21-stack/multiagent` (ветка `main`, с 2026-06-24). Изменения уровня хаба (hub-копии календарей `docs/calendars/*`, скиллы `.claude/skills/*`, docs/стандарты, реестр) — **коммитить и пушить в этот репо** по той же дисциплине, что и сайты. В коммит хаба НЕ попадают (gitignore): `sites/` (у каждого свой репо), `node_modules/`, `dist/`, `.astro/`, корневые scratch `_*`. Авто-память — отдельно (`~/.claude/.../memory/`, вне репо).

## Источники правды
- `docs/SITES-REGISTRY.md` — какие сайты, репо, домены, гео, состояние.
- `docs/STANDARDS.md` — стандарты контента/фото/новостей.
- `docs/MASTER-PLAN.md` — этапы, порядок, статус. Свериться перед любым шагом.
- `docs/audit/` — отчёты состояния.

## Ключевые правила (общие для всех 5 сайтов)
1. **Гео:** Грузия — ru/uk (СНГ) + en отдельным треком; остальные 4 — EN-first (Tier-1 Запад), ru/uk вторичны.
2. **Объём:** статья 1200–2000 слов; новость 1500–2000 знаков.
3. **Фото (минимум):** статья ≥5, новость ≥2, маршрут — фото каждой остановки, город ≥10, столица ≥15. Каждое фото уникально, своё или CC/PD с атрибуцией.
4. **Город = статья `category=goroda`** (коллекция `cities` не используется).
5. **Новости:** раздел хранит 10 дней, блок на главной — 2 дня (механика уже реализована во всех репо). `/news` НЕ слепой автопостинг: сбор → фильтр → превью владельцу → «ок» → публикация. Даты — только из источника текущего года.
6. **Достоверность:** факты только из источника, не выдумывать; кросс-сайтовый фактчекер проверяет даты/место/год/реальность перед публикацией.
7. **Дисциплина и синхронность доков:** после каждой доработки И перед каждым коммитом вся документация должна быть **актуальной** — `PROGRESS/HANDOFF/AUDIT`, **календари** (`KALENDAR.md` + hub-копия `docs/calendars/*`), **планы** (`docs/ROADMAP.md`, `docs/MASTER-PLAN.md`, backlog). Тронул контент/слот/план — тем же коммитом отметь это в календаре и роадмапе. Коммитить и пушить **без напоминаний** (и в репо сайта, и в **репо хаба** `sashakobtsev21-stack/multiagent`). Цель: после любого коммита всё в доках сходится с реальностью.
8a. **Понятные сообщения коммитов:** commit-message — человекочитаемый: что сделано + какой сайт/раздел + тема (напр. `publish(montenegro): Best Beaches 2026 article (en+ru)`), а не `fix`/`update`. Дашборд показывает названия коммитов — они должны объяснять активность без открытия диффа. См. память `descriptive-commit-messages`.
8. **Порядок этапов (см. MASTER-PLAN):** тех-аудит и чистка → SEO-аудит → контент-план по дням → минимальный контент → оркестрация → тест. Не забегать вперёд, чтобы поздние этапы не пришлось переделывать.

## Резервная схема работы (failover)
Когда основной исполнитель (Claude Code) недоступен/упёрся в лимиты — работа продолжается **резервной схемой**: владелец прогоняет `/work` и доработки через **Cursor с другой моделью ИИ**, по тем же скиллам, стандартам (`docs/STANDARDS.md`), календарям и гейтам (`npm run qa` зелёный, факты из источника, языки по сайтам, партнёрки только `/go/`). Правила одинаковы независимо от исполнителя.
- При возврате основного исполнителя — **сверить и примирить**: прочитать коммиты резервной схемы, проверить корректность (CI/qa, доки, календари), отметить опубликованное, при расхождениях — поправить. Не верить логу на слово (память `verify-repo-state-independently`).
- Резервная схема обязана так же синхронно обновлять доки/календари/планы и писать понятные коммиты (правила 7, 8a).

## Команды (по мере готовности, Этап 5)
- `/news [страны] [N]` — собрать новости по сайтам, один общий дайджест на согласование, после «ок» — публикация во все.
- `/content [страны] [дата]` — взять темы из по-дневного календаря каждого сайта и опубликовать статьи.
